# 数据面迁移上线观察清单

本文用于上线或灰度期间观察 netunnel 当前的数据面迁移状态。

当前策略：

- TCP：走 `data session + substream`
- `http_host`：走 `data session + substream`

## 上线前检查

1. 确认 agent 与 server 都已更新到包含 data session 的版本。
2. 确认服务端 `bridge_listen_addr` 与桌面端/agent 使用的 bridge 地址一致。
3. 确认服务端和 agent 都能正常启动，没有编译产物缺失。

## 第一阶段观察

直接观察纯新数据面的运行状态。

### 服务端 TCP

关注日志：`tcp runtime summary`

重点字段：

- `data_session_successes`
- `data_session_failures`
- `data_session_acquire_failures`
- `idle_timeouts`

判断建议：

- `data_session_successes` 持续增长：说明 TCP 正在走新数据面
- `data_session_failures` 很高：说明新数据面还有明显问题
- `data_session_acquire_failures` 很高：说明服务端在建立 data stream 这一层失败较多

### 服务端 data session

关注日志：

- `data session summary`
- `data session counters`
- `data session per-agent streams`
- `data session per-tunnel streams`

重点字段：

- `open_successes`
- `open_failures`
- `open_timeouts`
- `remote_closes`
- `local_closes`
- `overflow_closes`

判断建议：

- `open_timeouts` 不高：说明开流链路没有明显卡死
- `overflow_closes` 不高：说明 stream 消费没有明显堆积
- 某个 tunnel 长期高位：重点排查该 tunnel 的本地服务或访问模式

### 服务端 HTTP

关注日志：`public http summary`

重点字段：

- `data_session_successes`
- `data_session_failures`
- `data_stream_failures`

判断建议：

- `data_session_successes` 增长：说明 `http_host` 正在走新数据面
### Agent

关注日志：`agent data session summary`

重点字段：

- `retries`
- `active_streams`
- `open_successes`
- `open_failures`
- `write_fail_closes`
- `context_closes`

判断建议：

- `retries` 不高：说明 data session 连接稳定
- `write_fail_closes` 不高：说明 agent 向 session 写数据没有明显失败
- `active_streams` 不长期堆高：说明流可以正常回收

## 异常时的处理建议

如果纯新数据面出现异常：

1. 优先查看 `data_session_failures`、`open_timeouts`、`write_fail_closes`
2. 确认是否是单个 tunnel 或单个 agent 长期高位
3. 根据日志判断是 session 不稳，还是本地服务本身异常

## 关注目标

迁移是否成功，不看单次成功，而看一段时间内：

- 新数据面成功次数是否持续增长
- 是否还出现“运行一段时间后网络像堵住”的现象
