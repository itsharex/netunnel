# 网跃通新内网穿透系统技术方案

## 1. 目标

网跃通不再基于 `nps` 原项目继续深度改造，而是新建自己的内网穿透系统，仅参考 `nps` 的少量底层实现思路。

新系统的产品目标：

- 不提供 Web 管理页面
- 提供桌面端管理程序
- 服务端仅保留必要能力
- 客户端尽量轻量、稳定、易部署
- 数据存储统一使用 PostgreSQL
- 支持按流量充值和按余额扣费

当前明确保留的核心功能只有两类：

- TCP 映射
- 域名映射（HTTP/HTTPS）

## 2. 为什么不在 nps 基础上继续深改

虽然 `nps` 已经具备完整的内网穿透能力，但它与当前目标差异较大：

- 原项目强依赖 Web 管理端
- 功能过多，包含大量当前不需要的协议和模式
- 配置、任务、管理界面耦合较深
- 存储方式是 JSON 文件，不适合后续计费与多用户运营
- 现有目录结构和入口设计偏旧，后续维护成本高

因此更合适的路线是：

- 新建自己的服务端、客户端、桌面端
- 有选择地参考 `nps` 的转发与连接处理实现
- 业务模型、数据库、鉴权、计费全部按新系统重新设计

## 3. 总体架构

系统拆分为三个独立部分：

### 3.1 服务端 `netunnel-server`

职责：

- 用户认证与授权
- Agent 注册与心跳维护
- 隧道配置下发
- TCP 映射调度与转发
- 域名映射调度与转发
- 流量统计
- 余额校验与扣费
- 审计日志记录

建议技术：

- Go
- PostgreSQL
- 可选 Redis（后续需要缓存、限流、分布式状态时再引入）

### 3.2 客户端 `netunnel-agent`

职责：

- 部署在用户本地机器
- 与服务端保持控制连接
- 接收隧道配置
- 建立本地服务到服务端的转发通道
- 上报流量、状态、错误日志

建议技术：

- Go
- 单二进制运行
- 支持 Windows / Linux

### 3.3 桌面端 `netunnel-desktop`

职责：

- 用户登录
- 管理 Agent
- 创建与编辑隧道
- 查看域名映射
- 查看在线状态、流量、余额、扣费记录
- 发起充值

约定：

- 基于你指定的 Tauri Vue 模版创建
- 桌面端作为管理入口，替代原先 Web 管理后台

## 4. 核心功能范围

### 4.1 TCP 映射

场景：

- SSH
- RDP
- 数据库端口映射
- 任意自定义 TCP 服务

最小能力：

- 一个公网端口映射到某个 Agent 下的本地地址
- 支持启用/停用
- 支持连接数统计
- 支持流量统计
- 支持余额不足时禁止新连接

### 4.2 域名映射

场景：

- 将某个域名映射到 Agent 内网的 HTTP 服务
- 支持 HTTP / HTTPS

最小能力：

- 一个域名绑定一个隧道
- 支持 Host 路由
- 支持回源地址配置，例如 `127.0.0.1:8080`
- 支持证书策略

证书策略建议分阶段：

- 第一阶段：只支持平台统一证书或手动上传证书
- 第二阶段：支持自动签发与续期

## 5. 通信模型

推荐采用“控制连接 + 数据连接”模型。

### 5.1 控制连接

由 Agent 主动连接服务端，用于：

- 注册
- 认证
- 心跳
- 配置同步
- 状态上报
- 请求创建数据通道

建议：

- 首版使用 TCP 长连接即可
- 消息体可先用 JSON
- 后续如果追求性能，再演进为更紧凑协议

### 5.2 数据连接

用于真正承载用户业务流量。

建议：

- 首版优先保证稳定，不急着做复杂多路复用
- TCP 映射可先一连接一转发
- 域名映射由服务端接受外部请求，再转发到 Agent

## 7. 计费设计

### 7.1 计费目标

用户可以先充值余额，系统根据实际流量扣费。

### 7.2 建议的第一版规则

- 按总流量计费
- 入站和出站合并计算
- 流量统计按小时聚合
- 计费按小时或按固定周期结算

### 7.3 第一版扣费策略

建议先做“准实时周期扣费”，不要一开始就做逐包实时扣费。

推荐流程：

1. Agent 或服务端统计每条隧道流量
2. 服务端按时间窗口聚合到 `traffic_usages`
3. 定时任务按价格规则计算费用
4. 写入 `account_transactions`
5. 更新账户余额
6. 如果余额不足，则禁止新的隧道流量进入

### 7.4 余额不足策略

建议第一版采用：

- 禁止新建隧道
- 禁止新连接进入
- 已建立连接可以根据配置选择立即断开或自然结束

建议默认行为：

- 不立即踢掉现有连接
- 只阻止新连接

这样用户体验更平滑，系统也更容易控制。

## 8. 桌面端设计范围

桌面端将替代 Web 管理后台。

第一版页面建议：

- 登录页
- 首页概览
- Agent 列表
- 隧道列表
- 新建 TCP 映射
- 新建域名映射
- 流量统计页
- 账户余额页
- 充值记录页
- 系统日志页

首版不建议做过多视觉复杂度，先以“能稳定运营”为主。

## 9. 与 nps 的复用策略

建议只复用以下类型内容：

- TCP 转发相关的底层思路
- Agent 与服务端桥接模型参考
- 部分网络连接处理代码

不建议继承的部分：

- Web 管理端
- JSON 文件存储
- 多余协议
- 旧的配置体系
- 历史兼容逻辑

结论：

- 参考 `nps`
- 不兼容 `nps`
- 不继续背负 `nps` 的产品结构

## 10. MVP 开发顺序

### 第一阶段：基础骨架

- 初始化 `netunnel-server`
- 初始化 `netunnel-agent`
- 初始化 PostgreSQL schema
- 建立 Agent 注册与心跳
- 完成最小认证体系

### 第二阶段：核心隧道能力

- 完成 TCP 映射
- 完成域名映射
- 完成 Agent 配置下发
- 完成连接状态与基本日志

### 第三阶段：计费与余额

- 完成流量统计
- 完成账户余额
- 完成手工充值
- 完成周期扣费
- 完成余额不足限制

### 第四阶段：桌面端接入

- 基于 Tauri Vue 模版创建桌面端
- 接入登录
- 接入 Agent 管理
- 接入隧道管理
- 接入流量与余额页面

### 第五阶段：运营能力

- 接入真实支付渠道
- 增加充值订单与回调处理
- 增加价格规则管理
- 增加审计与风控能力

## 11. 当前建议结论

当前项目建议正式转向：

- 放弃在 `nps` 上持续深改
- 新建网跃通自己的服务端、客户端、桌面端
- 只保留 TCP 与域名映射两类核心功能
- 从第一天开始按 PostgreSQL + 余额计费模型设计
- 后续桌面端基于指定 Tauri Vue 模版创建

这条路线更适合做一个真正可持续演进、可运营、可计费的自有产品。
